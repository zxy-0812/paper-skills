# ABOUTME: 聚焦 /root/git/drivestudio 与相关 3DGS 框架，实现可运行、可实验的代码修改

# project-to-code

本 Skill 负责在**已有创新点工程化文档**（例如 `docs/relighting-project.md`）与**已有框架/代码结构文档**（如 `docs/ARCHITECTURE.md`）的前提下，
系统化地将文档中描述的工程实现步骤，落实为 `/root/git/drivestudio` 中可运行、可实验的**实际代码改动**，并在过程中维护一份**代码实现总结文档**。

典型使用场景示例：
- “创新点工程化转换为代码”
- “根据 relighting-project.md 改 drivestudio 里的代码”
- “把工程实现文档里这些步骤真正落成代码实现”
- “在 ARCHITECTURE 指导下修改 drivestudio 代码实现这些创新点”

---

## 总体工作流

当此 Skill 被触发时，遵循以下总体步骤执行。默认目标工程为 `/root/git/drivestudio`，如用户指定其他工程路径则以用户为准。

1. **收集输入与上下文**
   - 必须确认以下输入是否可用：
     - **创新点工程化文档**：例如 `docs/relighting-project.md` 或用户指定的其它工程实现文档；
     - **框架/代码结构文档**：例如 `docs/ARCHITECTURE.md`，用于理解模块划分、数据流、训练/推理流程；
     - **目标代码目录**：默认 `/root/git/drivestudio`，或用户提供的其他工程路径。
   - 使用 `Read` 工具：
     - 读取创新点工程化文档全文；
     - 读取 `docs/ARCHITECTURE.md`；
     - 根据工程化文档中提到的关键模块/文件名，必要时在 `/root/git/drivestudio` 中读取对应代码文件。

2. **梳理“工程化文档 → 框架结构 → 代码位置”的映射**
   - 通读工程化文档，重点围绕以下信息建立映射表：
     - 文档中每个**模块 / 子系统 / 步骤**对应的：
       - 在 `docs/ARCHITECTURE.md` 中的模块或子系统名；
       - 在 `/root/git/drivestudio` 中的**目录/文件/类/函数**（根据架构文档与实际代码结构推断）。
   - 对每一个实现步骤，回答以下问题：
     1. 该步骤属于哪一个模块（渲染、训练、评估、数据处理等）？
     2. 在 `docs/ARCHITECTURE.md` 中对应哪一节或哪组组件？
     3. 在 `/root/git/drivestudio` 实际代码中应修改/新增哪些文件、类、函数？
   - 将此映射整理为一份内部表格（可在总结文档中以 Markdown 表格形式记录），为后续按模块改动提供路线图。

3. **按模块、按步骤地规划代码改动顺序**
   - 基于工程化文档的章节结构和改动路线，将实现过程划分为若干模块/阶段，例如：
     - 高斯定义/属性扩展相关改动；
     - 渲染管线（光照建模、着色、遮挡处理等）相关改动；
     - 训练管线（损失项、采样策略、scheduler 等）相关改动；
     - 数据预处理或数据加载改动；
     - 评估与可视化脚本改动。
   - 对每个模块，明确：
     - **当前阶段要完成的最小可行改动集（MVP）**；
     - 若工程化文档中的计划较大，则将其拆解为可迭代的子阶段，并记录未来阶段的规划。

4. **执行代码改动时的通用原则**
   - **函数与类设计**
     - 生成/修改的函数与类应满足：
       - 命名清晰、符合项目命名风格（参考项目现有实现）；
       - 有完整的文档字符串（docstring）说明输入、输出、行为与边界条件；
       - 包含必要的参数校验与错误处理；
       - 遵循当前语言与框架（通常为 Python + PyTorch）的最佳实践。
   - **模块与包结构**
     - 优先复用现有模块结构；仅在确有必要时创建新模块或子包；
     - 导入路径应保持简洁与稳定，避免循环依赖；
     - 设计清晰的模块 API（对外暴露的类/函数），将内部实现细节封装起来。
   - **性能与显存占用**
     - 对可能增加显存占用的改动（新增特征通道、更大网络层、额外 buffer 等）：
       - 在实现时显式考虑 batch size、feature 维度、数据类型（如 FP16/FP32）、稀疏性等；
       - 在总结文档中记录潜在显存风险点及可选的降显存策略（例如使用 checkpointing、减少冗余缓存等）。
   - **错误处理与边界检查**
     - 对关键接口（如数据加载、主训练入口、渲染循环）中的新逻辑：
       - 增加参数范围检查（如非负、区间约束、形状校验）；
       - 对可能失败的外部依赖（文件路径、网络请求等）增加合理的异常信息；
       - 保证报错信息与项目现有风格一致，便于调试。

5. **依赖管理与 `requirements.txt` 更新**
   - 在阅读工程化文档和规划改动时，识别是否需要引入额外 Python 包或工具：
     - 新的渲染/数值库；
     - 额外的可视化、日志、评估工具；
     - 与相关论文/开源代码一致的特定依赖。
   - 对任何新引入的依赖，执行以下规范：
     - 在项目根目录的 `requirements.txt` 中追加对应包名及合适版本范围（若现有依赖中已包含，则检查是否需要版本升级）；
     - 若版本选择有不确定性，在总结文档中记录“依赖版本决策”的说明及可替代版本。

6. **逐阶段执行代码改动并维护总结文档**
   - 在每一个实现阶段：
     - 使用 `Read` 和 `ApplyPatch` 对应的代码文件；
     - 以最小可行改动为原则，完成当前阶段的实现；
     - 在总结文档中记录本阶段改动内容，格式见下文“总结文档结构”。
   - 对于**较为复杂或跨模块的大改动**：
     - 可以分成多个子阶段（例如“仅添加数据结构与接口，不启用新路径”、“启用新路径但保留旧分支作为备份”、“完全切换到新实现”）；
     - 每个子阶段完成后，在总结文档中更新当前状态，并记录后续阶段的具体计划。

7. **处理工程化文档中含混不清或无法直接转化为代码的部分**
   - 对于工程化文档中描述不够清楚、无法直接映射到具体代码改动的条目：
     - 不要臆造实现方式；
     - 在总结文档中单独开辟小节记录这些条目，并对每条添加：
       - 原始描述片段；
       - 当前无法落地为代码的原因（例如依赖未公开资源、论文描述不足、与现有架构冲突等）；
       - 暂时的处理策略（如“保持现有实现不变”、“保留占位接口但不启用”）；
       - 建议的后续行动（进一步阅读论文、寻找开源实现、与人讨论等）。

## 代码实现与结构要求（实践层面的细则）

在具体生成或修改代码时，需遵守以下结构与质量要求：

1. **导入部分**
   - 导入语句按项目现有规范排序与分组（标准库、第三方库、本地模块）；
   - 避免使用 `import *`，保持显式导入；
   - 尽量避免在函数内部重复导入，除非确有懒加载需求（此时在 docstring 中注明原因）。

2. **文档字符串与注释**
   - 所有对外暴露的函数与类必须有完整文档字符串，说明：
     - 功能简述；
     - 参数列表与类型（如适用，使用类型提示）；
     - 返回值含义与类型；
     - 可能抛出的异常及触发条件；
     - 特殊约定或性能/显存注意事项。
   - 仅在必要时添加行内注释，用于解释非显而易见的设计决策或特殊处理逻辑，避免冗余的“翻译式”注释。

3. **实现代码（核心逻辑）**
   - 遵循工程化文档与架构文档的设计意图，实现：
     - 完整的控制流与数据流；
     - 对关键分支与边界情况（如空 batch、单视角、异常几何等）的处理；
     - 对已有接口的兼容性（尽量不破坏原有调用方或提供兼容层）。
   - 进行必要的性能思考：
     - 评估是否可以复用已有张量或中间结果，避免重复计算；
     - 对可并行化部分使用向量化或批处理；
     - 在需要时考虑使用 FP16/混合精度，并记录潜在的数值稳定性问题。

4. **测试代码与使用示例（如需要/如工程已有模式）**
   - 若工程已有单元测试框架（pytest 等）：
     - 为关键新逻辑添加单元测试或扩展现有测试用例；
     - 测试至少覆盖：
       - 正常路径；
       - 典型边界情况或错误输入；
       - 与旧实现对比的关键行为（如输出维度、数值范围、是否报错等）。
   - 若工程已有示例脚本或 notebooks：
     - 可选地添加/修改示例，将新功能纳入典型使用流程；
     - 在总结文档中记录这些示例文件路径与预期行为。

## 总结文档结构与维护要求

在执行此 Skill 时，必须维护或创建一份**代码实现总结文档**（可以是新的 Markdown 文件，或在现有工程化文档基础上增加“代码实现追踪”章节），建议结构如下：

```markdown
# 代码实现总结：<创新点主题>

## 1. 总览
- 目标工程：`/root/git/drivestudio`
- 参考工程化文档：`docs/relighting-project.md`
- 参考架构文档：`docs/ARCHITECTURE.md`

## 2. 模块级改动概览
- 模块 A（例如：渲染管线 / 光照建模）
  - 相关文件：`...`
  - 主要改动：新增类 Xxx，修改函数 Yyy，引入损失 Zzz
- 模块 B（例如：训练管线）
  - ...

## 3. 按步骤记录的具体改动
### 3.1 阶段/步骤 S1：<名称或对应用工程化文档的小节>
- 改动目标：
  - ...
- 具体代码改动（概要，不必粘贴完整代码）：
  - 修改 `file_a.py` 中类/函数：Xxx
  - 新增 `file_b.py` 中函数：Yyy
- 依赖与配置变更：
  - `requirements.txt`：新增/修改包 ...
  - 配置文件：...
- 显存与性能注意点：
  - ...
- 当前状态：
  - [x] 已完成实现与基本自检
  - [ ] 已通过完整训练/实验验证（如未完成则保留 TODO）

### 3.2 阶段/步骤 S2：...
...

## 4. 含混或暂无法落地为代码的条目
- 条目 1：
  - 原始描述：...
  - 无法落地原因：...
  - 当前处理策略：跳过/保留接口/待进一步确认
  - 建议后续行动：...

## 5. 后续规划
- 下一步计划的改动阶段与大致内容
- 需要进一步验证或阅读的外部资源（论文、开源仓库等）
```

在整个实现过程中，应持续更新该总结文档，确保：
- 已完成的改动都在文档中有记录，便于后续排查和协作；
- 未完成或待确认的部分明确标记，避免误以为已经完全实现。

## 与其他 Skill 的协同

- **与 `innvoation-to-project` Skill 的关系**
  - `innvoation-to-project` 负责：从**创新点总结文档**与 `docs/ARCHITECTURE.md` 出发，生成**工程实现说明文档**；
  - `project-to-code` 负责：在工程实现说明文档已经存在的前提下，将文档内容落实为 `/root/git/drivestudio` 中的**实际代码改动与实现总结**。
  - 若当前对话中尚未有高质量的工程实现文档，可优先调用 `innvoation-to-project` 生成/完善后，再使用本 Skill 进行代码层面的落地。

- **与 `code-architechure`、`read-arxiv-paper` 等 Skill 的关系**
  - 当工程化文档中引用的论文或开源代码实现对具体代码结构有重要影响时：
    - 可使用 `read-arxiv-paper` 深入理解对应论文的细节；
    - 可使用 `code-architechure` 对外部仓库进行结构化分析，并将有价值的实现思路迁移到本项目；
    - 在迁移或参考实现时，遵循本 Skill 中的代码质量与结构要求，并在总结文档中记录来源与差异。

## 输出要求

在用户请求使用本 Skill 时，回答应包含以下内容（根据任务阶段灵活取舍）：

1. **改动规划与路线图**
   - 清晰说明本次准备完成的阶段/模块；
   - 简要列出涉及的主要文件、类、函数。

2. **实际代码实现（以补丁或片段形式展示）**
   - 对每个阶段提供具体的代码修改内容（或补丁摘要）；
   - 避免在回复中粘贴超大文件的全部内容，仅展示关键改动片段。

3. **总结文档的更新或新增内容**
   - 输出或更新“代码实现总结文档”的相关章节；
   - 标注已完成与待完成的部分。

4. **依赖与显存注意事项**
   - 明确指出新增/修改的依赖包及其版本（若有）；
   - 对可能影响显存或性能的改动给出简要说明与建议。

在无法将某些工程化描述直接落地为代码实现时，务必在总结文档和回复内容中**显式说明原因并标记为 TODO/待确认**，而不是给出缺乏依据的实现代码。
