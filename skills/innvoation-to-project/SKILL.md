# innvoation-to-project

本 Skill 负责在**已有创新点总结文档**与**已有框架/代码结构文档**（如 `docs/ARCHITECTURE.md`）的前提下，
生成一份**工程实现说明文档（Markdown）**，目标是让用户可以据此直接在 `/root/git/drivestudio`中实现对应创新点并开展实验，尽可能消除实现层面的模糊与困惑。

使用场景触发示例：
- “从创新点总结生成工程实现文档”
- “帮我把 relighting 创新点写成可落地的实现方案”
- “把 @relighting.md 里的创新点转成 OmniRe 里要改哪些模块”
- “根据 ARCHITECTURE.md 生成实现这些创新点的工程说明”
- “对这份工程实现文档我有几个疑问……”
- “请根据我的建议修改 relighting-project.md”

---

## 总体工作流

当此 Skill 被触发时，先判断用户意图：

- **若用户提供的是对已有工程实现文档的疑问或修改建议**（例如附上了 `docs/relighting-project.md` 并列出问题/建议）→ 执行 **工程实现文档的评审与修订（Review）** 流程（见下节），可跳过或仅做轻量化的“收集输入与上下文”。
- **若用户是首次请求生成工程实现文档** → 按以下步骤 1–6 顺序执行。

1. **收集输入与上下文**
   - 确认以下信息是否已在对话或 workspace 中给出：
     - **创新点总结文档**（例如 `.cursor/knowledge/relighting.md` 等）
     - **框架/代码结构文档**：例如 `docs/ARCHITECTURE.md`，或者其他 OmniRe / DriveStudio 相关架构文档
   - 使用 `Read` 工具：
     - 读取创新点总结文档的完整内容
     - 读取 `docs/ARCHITECTURE.md`（如存在），了解整体模块划分、数据流、训练/推理流程
   - 若有多个创新点文档或多个架构文档：
     - 先快速浏览标题/小节，建立**创新点 → 模块/子系统**的粗粒度映射

2. **识别“实现不够清楚”的描述**
   - 在创新点总结文档中，重点查找以下模式的表达：
     - 含糊的资源/接口引用，如：
       - “公开材质库与经验参数范围”
       - “使用现有 relighting 模块”
       - “借鉴某某论文的损失函数/网络结构”
     - 未说明**具体工程实现位置**的句子，例如：
       - “在感知模块中加入 XXX”
       - “在渲染阶段叠加一个光照校正分支”
     - 没有给出**具体参数/数据结构/调用时机**的句子
   - 对每一处模糊描述，至少回答以下三个问题：
     1. **它指的是什么资源/能力？**
        - 根据上下文先抽象出“需要什么信息”：是算法/损失/渲染策略，还是数据预处理、材质/几何先验、训练设置等。
        - 不要预设一定是“材质库”或“经验参数库”，而是紧贴当前创新点文档里的表述来归纳。
     2. **应该从哪里寻找这些资源或参数？**
        - 优先策略（基于你当前项目的知识库布局）：
          - 第一步：在**创新点总结文档**中查找对应的“论文总结文档名称”，例如：
            - 文本中出现 `material-informed-gaussian-splatting-digital-twin`，
            - 则其对应的论文总结文档路径为  
              `docs/papers/summaries/material-informed-gaussian-splatting-digital-twin-summary.md`。
          - 第二步：根据论文总结文档名称，在 `~/.cache/knowledge/` 中查找对应的**论文原文目录**：
            - 所有创新点参考论文的原文都已缓存于 `~/.cache/knowledge/`；
            - 例如上述论文的原文目录为 `~/.cache/knowledge/2511.20348/`。
          - 第三步：在“论文总结文档 + 论文原文”中查找：
            - 具体的算法说明、损失定义、渲染/训练管线描述；
            - 需要的参数、先验、数据格式或实现细节。
          - 如有必要，再使用 `WebSearch`/`WebFetch` 作补充：
            - 论文名 + “github”
            - 论文名 + “code”
            - 论文名 + “project page”
     3. **在当前 OmniRe / DriveStudio 工程中，应该落到哪些模块或文件？**
        - 结合 `docs/ARCHITECTURE.md` 与 DriveStudio/OmniRe 的实际定位：
          - 该项目是一个 3DGS 渲染框架，核心由**不同类型的高斯定义**、**渲染管线**与**训练管线**构成；
        - 将模糊描述优先映射到：
          - 高斯参数/属性定义及其更新逻辑（如外观、密度、位姿等）；
          - 渲染管线中的光照建模、着色、可见性/遮挡处理等位置；
          - 训练管线中的损失项、正则项、采样策略、学习率/调度相关配置。

3. **寻找可参考的开源代码并写入文档**
   - 对每一个创新点条目，执行以下步骤：
     1. 在创新点总结文档中找出**关联论文**：
        - 通常在引用列表或小节标题/括号内给出，如 “[Nighttime PBR Gaussian Splatting]” 等；
        - 同时定位其对应的**论文总结文档**路径（如 `docs/papers/summaries/...-summary.md`）。
     2. 优先阅读对应的**论文总结文档**：
        - 其中往往已经整理了“是否有开源代码”、“项目主页/仓库链接”等信息；
        - 若总结文档中已给出开源与否的结论，应直接在工程实现文档中复用该信息。
     3. 若总结文档未给出开源信息或需要进一步确认：
        - 使用 `WebSearch` 搜索对应论文是否有开源代码：
          - 关键词组合：
            - 论文标题 + “github”
            - 论文标题 + “project page”
            - 第一作者名 + 论文关键词
     4. 如果找到开源代码：
        - 在工程实现文档中记录：
          - **仓库名称**
          - **简要说明其和本创新点的关系**（例如：提供了 nighttime relighting 训练 pipeline，或提供了高斯参数优化策略）
          - **仓库链接**（使用 Markdown 链接格式）
        - 如果仓库结构清晰且与本项目接近，可进一步：
          - 通过后述的 `code-architechure` Skill（代码架构分析 Skill）进行结构级阅读与映射；
          - 在实现文档中写出“可直接参考的函数/模块名称 + 在我们项目中对应的实现位置”。
     5. 如果暂未发现开源代码：
        - 在文档中明确说明 “未找到官方开源实现”，并：
          - 复用论文总结文档中的结论（若已有说明）；
          - 或根据论文伪代码 / 算法描述给出伪实现方案。

4. **基于论文与代码进一步细化实现细节**
   - 对于仍然不够清晰的实现说明，采用以下路径：
     - **通过 `read-arxiv-paper` Skill 继续深入阅读论文原文**：
       - 当创新点总结给出论文引用但实现细节不足时：
         - 利用前述的 `~/.cache/knowledge/<arxiv-id>/` 路径，锁定对应论文原文；
         - 调用 `read-arxiv-paper` Skill，使用其中的 Pass3 阅读方式，聚焦以下内容：
           - 高斯定义、渲染公式、光照/材质建模方式等与 3DGS 直接相关的部分；
           - 损失函数定义与优化目标；
           - 训练/推理管线步骤与伪代码；
           - 实验设置中与数据预处理 / 光照建模 / 先验约束相关的具体参数。
       - 将这些内容总结成：
         - “需要在我们工程中**新增/修改哪些高斯属性/渲染或训练管线模块**”
         - “关键超参数/先验参数范围是什么”
     - **通过 `code-architechure` Skill 对开源代码进行结构级的阅读与映射**：
       - 当已经找到相关开源代码仓库时：
         - 使用 `code-architechure` Skill，对该仓库进行系统化架构分析；
         - 关注其如何组织 3DGS 相关模块、高斯属性、渲染与训练管线；
       - 将分析结果映射为：
         - “对应我们项目中的哪个子系统（高斯定义、渲染、训练、评估等）”
         - “最小可行集成路径（只引入子模块或迁移部分代码）”；
         - 以及在 `/root/git/drivestudio` 中建议修改/新增的文件与类/函数名称。

5. **生成工程实现文档（Markdown）结构**
   - 针对每个创新点/子创新点，按照统一模板输出，建议整体结构如下：

   ```markdown
   # 工程实现文档：<创新点主题，例如 Relighting>

   ## 1. 适用工程范围与前置依赖
   - 目标工程：`/root/git/drivestudio`（OmniRe 子系统）
   - 参考架构文档：`docs/ARCHITECTURE.md`
   - 相关创新点总结文档：`.cursor/knowledge/relighting.md` 等

   ## 2. 创新点分解与实现位置映射
   ### 2.1 创新点 A：<名称>
   - 摘要：从创新点总结文档中摘取 1–2 句核心描述
   - 当前描述中不清楚的点：
     - 问题 Q1 / 描述不清位置
     - 问题 Q2 / ...
   - 进一步澄清与决策：
     - 对 Q1 的澄清与工程化解释
     - 对 Q2 的澄清与工程化解释
   - 在当前工程中的落点：
     - 相关模块/文件（基于 `docs/ARCHITECTURE.md`）
     - 推荐修改入口（函数/类/脚本）

   ### 2.2 实现依赖与可复用代码
   - 参考论文：
     - [论文名 1](链接) —— 与本创新点的关系说明
   - 开源实现：
     - [Repo 名称](GitHub 链接)：对应的模块/函数说明
     - 如果无官方 repo：记录“暂无官方开源实现”，并给出可替代参考

   ## 3. 具体实现步骤（面向 OmniRe）
   - Step 1：在 `drivestudio` 中定位/创建 Xxx 模块
   - Step 2：按照论文/开源实现引入 Yyy 结构或损失
   - Step 3：在训练脚本中新增/修改配置项 Zzz
   - Step 4：在评估/可视化脚本中添加新的指标或渲染结果导出

   ## 4. 参数与资源配置
   - 数据集预处理改动（例如需要生成额外的 PBR 材质标签）
   - 重要超参数与推荐初值

   ## 5. 预期风险与验证实验
   - 风险点（训练不收敛、材质先验不准确等）
   - 最小验证实验设计（例如在 Waymo 子集上进行 ablation）

   ## 6. 待解决问题（可选，Review 时维护）
   - 当前无法从文档/论文/代码得出确定结论的疑问或分歧；
   - 每条建议包含：问题简述、暂无法确定的原因、建议的后续动作。
   ```

   - 实际生成时，需将模板中的占位符替换为基于当前创新点文档与架构文档、论文/开源代码分析得到的**具体内容**。

6. **确保“没有明确困惑未被提及”**
   - 在完成工程实现文档草稿后，显式检查并补充：
     - 是否还有**实现位置模糊**的地方（例如仅写“修改渲染模块”但未指明文件/函数）
     - 是否还有**资源来源不清楚**的地方（例如仅写“公开材质库”但未列出具体候选）
     - 是否还有**参数/超参数未给出范围或推荐值**的地方
   - 若存在上述模糊点：
     - 在文档中使用 “**问题待定**” 或 “**TODO**” 小节明确列出；
     - 给出下一步可以采取的动作（例如“进一步阅读论文 X 第 Y 节”、“尝试在 GitHub 中以关键词 Z 搜索”）。

## 工程实现文档的评审与修订（Review）

当用户在对**已生成的工程实现文档**提出疑问或修改建议时，按本流程处理。

### 输入与前置

- 用户提供的**工程实现文档**路径（如 `docs/relighting-project.md`），以及：
  - **疑问**：对文档中某条描述不理解、或希望得到更细的解释；
  - **建议**：希望增加/删除/改写某段内容，或调整实现顺序、参数、依赖等。
- 可选：用户同时引用的创新点总结文档、架构文档，便于对照评估。

### 执行步骤

1. **读取当前工程实现文档与相关上下文**
   - 使用 `Read` 读取目标工程实现文档全文；
   - 若用户未指定，可根据文档内“适用工程范围与前置依赖”一节推断相关创新点文档与 `docs/ARCHITECTURE.md`，必要时读取以便评估建议与解答疑问。

2. **逐条处理用户的疑问**
   - 对每个疑问：
     - 在工程实现文档、创新点总结文档、论文总结文档或论文原文（`~/.cache/knowledge/`）中查找可支撑的表述；
     - 若能找到明确依据：给出**简洁解答**，并注明依据来源（如“见文档 §X”、“见论文总结 Y 中……”）；
     - 若当前文档表述不清导致疑问：在解答的同时，**顺带修订文档**中对应段落，使后续读者不再产生同一疑问；
     - 若无法从现有资料得出确定结论（例如涉及未实现细节、或与框架行为需实验验证）：**不要强行作答**；将该条列为**待解决问题**，写入文档的“待解决问题”小节（见下），并简短说明“为何暂时无法回答”以及“建议的后续动作”（如阅读某论文某节、在代码中做小实验验证等）。

3. **逐条评估用户的建议**
   - 对每条建议判断**合理性**：
     - 是否与创新点总结文档的目标一致；
     - 是否与 `docs/ARCHITECTURE.md` 及 DriveStudio 的 3DGS 架构（高斯定义、渲染管线、训练管线）相容；
     - 是否与所引论文/开源代码的常见做法不冲突（若文档中已引用）。
   - **合理且可落实的建议**：在工程实现文档中**直接采纳**，修改对应章节（如实现步骤、参数与资源配置、实现位置映射等），并在修订处或文档末尾的“修订说明”中简要记录“根据用户建议：……”。
   - **部分合理但需折中的建议**：在文档中采纳可采纳的部分，对存在争议或多种方案的部分，在文档中注明“可选方案：A / B，当前建议为 A，理由……”，或将该分歧点列入“待解决问题”供后续决定。
   - **不合理或与现有依据明显冲突的建议**：不修改文档以迎合该建议；在回复中**礼貌说明不采纳的理由**（引用文档/论文/架构依据），并可选地在文档“待解决问题”中记一条“用户曾提出 X，经评估因 Y 未采纳，若后续有 Z 类证据可再考虑”。

4. **维护“待解决问题”小节**
   - 在工程实现文档中保留或新增 **“## 待解决问题”**（或 “**问题待定 / TODO**”）小节；
   - 将本次评审中**无法确定回答的疑问**、**需后续实验或阅读才能决定的建议**统一列在该小节中，每条格式建议为：
     - 问题或议题的简短描述；
     - 当前无法确定的原因（如“论文未给出”、“需在 Waymo 上做 ablation 才能确定”）；
     - 建议的后续动作（如“阅读 XXX 论文 §Y”、“在 drivestudio 中先做最小实验验证”）。
   - 这样既保证文档可被持续改进，又避免给出没有依据的断言。

### 输出要求（Review 场景）

- 回复中先**逐条回应**用户的疑问与建议（采纳/部分采纳/不采纳及理由）；
- 再给出**修订后的工程实现文档**（或仅给出修订过的章节/片段），并标明“待解决问题”小节的最新内容；
- 若某条被列为待解决问题，在回复中明确提示用户：“已写入文档‘待解决问题’小节，建议后续……”

## 使用与输出要求

- **生成文档场景**：回答用户时：
  - **直接输出整理好的工程实现 Markdown 文档**（可以是完整文档或草稿版本）；
  - 避免只给出零散建议，而应给出结构化章节；
  - 在文档中清晰区分：
    - “已明确的实现步骤/参数”
    - “仍存在不确定性的部分及后续行动建议”。
  - 如有必要，可在最后给出一段简短说明，提示用户下一步可从哪些模块入手改代码、哪些小节需在实际试验中进一步修正与补充。
- **评审与修订（Review）场景**：按“工程实现文档的评审与修订”一节的输出要求执行——先逐条回应疑问与建议，再给出修订后的文档或修订片段，并维护“待解决问题”小节；难以回答的条目只写入待解决问题，不强行作答。
